## 2026-03-14：Q&A 交互方式

## 2026-03-15：PDF 处理策略（已推翻旧决策）

**决定**：App 服务端处理文件转换。用户上传 PDF，系统自动提取文本，用户无需手动转换。Phase 1 使用 `pdf-parse` 库，扫描版 OCR 作为后续增强。

**理由**：让用户自己转换文件是把系统内部问题推给用户，产品体验差。好的产品应该隐藏复杂度。旧决策（要求用户用外部工具预处理）被推翻。

**状态**：已关闭

**Keywords**: PDF处理, pdf-parse, OCR, 服务端处理, 文本提取

## 2026-03-18：Phase 2 产品方向——PDF 阅读器 + 截图问 AI

**决定**：Phase 1.5 质量修复暂停，转向 Phase 2。核心功能：用户上传 PDF 后立刻可在浏览器内看 PDF + 截图区域问 AI。OCR 后台运行，完成后解锁结构化学习。

**理由**：扫描版 PDF OCR 需要 12 分钟，新用户等不了。必须让用户上传后立刻有事可做（看 PDF + 问 AI），把等待时间变成使用时间。

**状态**：已关闭

**Keywords**: Phase 2, PDF阅读器, 截图问AI, OCR, 用户体验

## 2026-03-18：截图 OCR 改为常驻 HTTP 服务

**决定**：截图问 AI 的 OCR 不再每次调用都启动新 Python 进程，改为常驻 HTTP 服务（`scripts/ocr_server.py`，端口 9876）。Node.js 通过 `http.request` 直连，绕过 `HTTP_PROXY`。

**理由**：PaddleOCR 模型加载需要 10-20 秒，每次 `execFile` 都重新加载，撞 30s 超时。常驻服务模型只加载一次，后续每次 OCR 推理约 3-5 秒。

**状态**：已关闭

**Keywords**: PaddleOCR, HTTP服务, ocr_server.py, 性能优化

## 2026-03-21：Q&A 反馈时机改为即时反馈

**决定**：Q&A 从"全部答完后批量反馈"改为"每题答完即时反馈"。产品不变量 #5 同步更新。

**理由**：(1) Ericsson 即时反馈原则支持逐题反馈效果更好；(2) 旧方案是 CC CLI 的技术限制导致的妥协，Web App 无此限制；(3) 即时反馈的短循环对 ADHD 用户更友好。

**可逆性**：容易反悔——改一个前端交互和一个 API 调用时机即可。

**状态**：已关闭

**Keywords**: Q&A, 即时反馈, 产品不变量, Ericsson

## 2026-03-21：MVP 重新设计——教练核心 + 阅读器

**决定**：重新定义 MVP 范围为"教练链路 + PDF 阅读器 + 独立问答通道"。教练链路基于 v2.0 学习系统 prompt，5 个 AI 角色分工（提取器/教练/考官/复习官/助手），KP 成为核心数据实体。不做翻译、书签、知识图谱等工具层功能。

**理由**：产品核心价值是"学习执行力外包"（教练），不是"更好用的学习工具"。先做教练核心，与竞品形成差异化定位。工具层功能后续叠加。

**设计文档**：`docs/superpowers/specs/2026-03-21-mvp-redesign-design.md`

**可逆性**：容易反悔——工具层功能随时可加，不影响教练核心架构。

**状态**：已关闭

**Keywords**: MVP, 教练核心, PDF阅读器, KP, AI角色分工

## 2026-03-21：目标用户定位——中国留学生

**决定**：MVP 目标用户为中国留学生中"想学但自驱不够"的群体——离开国内中学和家长约束后缺乏结构化学习能力，需要外力推动。

**理由**：(1) 与产品创始人自身痛点一致；(2) 有付费意愿和能力；(3) "教练"定位对这群用户价值最高；(4) 竞品的"工具"定位无法解决他们的核心问题。

**可逆性**：容易反悔——用户定位是营销和优先级问题，不影响产品架构。

**状态**：已关闭

**Keywords**: 目标用户, 中国留学生, 用户定位, 教练, 竞品差异化
