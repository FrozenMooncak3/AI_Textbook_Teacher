# 调研 4：教学 UX — 怎样让学不无聊

**调研日期**: 2026-04-11
**用途**: 教学系统设计——解决"一次教 3-5 个 KP × 每个 3-6 轮对话 = 用户怎么坚持"
**前置**: 调研 1-3

---

## 核心问题

教学系统加入后，一个模块的学习可能包含 10+ 个 KP，每个 KP 需要 2-6 轮教学对话。即使只教 3-5 个 KP 一次，也是 10-25 轮对话。

**如果体验像"回答 25 个问题"，用户会在第 5 个走掉。**

---

## Duolingo 的启示：什么让学习上瘾

Duolingo 4770 万 DAU（2025），用户留存从 12% 提升到 55%。他们做对了什么：

### 有效的设计模式

| 模式 | 机制 | 效果数据 |
|------|------|---------|
| **条纹 (Streak)** | 连续学习天数计数器，断了就清零 | 维持 7 天条纹的用户长期活跃率 3.6 倍 |
| **进度可视化** | 每课完成后进度条填充，有明确的"终点" | 课程完成率提升 50%（Colorado 大学研究） |
| **微学习** | 每个 lesson 3-5 分钟，不是 30 分钟 | 用户更愿意"再做一个" |
| **即时反馈** | 答对绿色闪烁+音效，答错红色+正确答案 | 减少挫败感，保持节奏 |
| **条纹冻结** | 花金币保住条纹（损失厌恶的安全阀） | 减少流失 21% |
| **XP + 排行榜** | 每次练习获得经验值，周排行榜 | 参与度提升 40% |

### 不适用于我们产品的模式

| 模式 | 不适用原因 |
|------|-----------|
| 排行榜 | MVP 阶段没有社交功能；教材学习不适合比赛 |
| 生命/心数 | 教材学习不是闯关，答错不应被惩罚 |
| 付费道具 | 商业模式不同 |
| 课程树/技能树 | 教材本身的章节结构就是学习路径 |

### 可借鉴的核心理念

**"让学不觉得在学"**——Duolingo 把学习伪装成游戏。我们不需要这么极端，但核心理念适用：**减少"我正在被教"的感觉，增加"我正在理解"的感觉**。

## 微学习原则：拆碎教学单元

### 关键数据

- 微学习最佳时长：5-15 分钟一个完整单元
- 工作记忆容量：4±2 个信息块（Miller 定律）
- Duolingo 每课：3-5 分钟
- 注意力自然衰退点：约 15-20 分钟

### 对我们的设计约束

| 参数 | 推荐值 | 理由 |
|------|-------|------|
| 每次教学 session 的 KP 数 | 3-5 个 | 工作记忆 4±2 块 |
| 每个 KP 的对话轮数 | 2-3 轮（轻量）/ 4-6 轮（完整） | 每个 KP 2-5 分钟 |
| 每次 session 总时长 | 10-20 分钟 | 微学习上限 15 分钟 + 缓冲 |
| 检查点间隔 | 每 3 个 KP | 约 10 分钟一个自然休息点 |

## 教学 UX 设计原则

### 原则 1：把 25 轮对话变成 5 个"小关卡"

**问题**：25 轮对话看起来无边无际。
**解法**：把 3-5 个 KP 分组为可见的进度段。

```
教学 Session: "第 3 章 · 供需理论"

  [KP 1: 需求定义] ████████░░  ← 你在这里
  [KP 2: 供给定义] ░░░░░░░░░░
  [KP 3: 均衡价格] ░░░░░░░░░░

  完成 KP 1 后显示：
  ✓ KP 1: 需求定义 — 掌握
  [KP 2: 供给定义] ████░░░░░░  ← 你在这里
  [KP 3: 均衡价格] ░░░░░░░░░░
```

**关键**：用户始终知道"我在哪"和"还剩多少"。终点可见 = 坚持意愿强。

### 原则 2：教学对话不要像审问

**问题**：连续追问"你知道什么？""为什么？""还有吗？" = 审问。
**解法**：混合呈现和互动，不要每轮都是问题。

**好的节奏**：
```
AI: [呈现] 需求定律说的是——价格上升，需求量下降。
    想象一下：你喜欢的奶茶从 15 块涨到 30 块，你是不是就少买了？

AI: [确认] 那如果奶茶降价到 5 块呢？你觉得你会怎么做？

用户: 多买

AI: [反馈+推进] 对，这就是需求定律的核心。
    不过有个反直觉的情况——有些商品涨价反而买得更多。
    你能想到什么例子吗？
```

**坏的节奏**：
```
AI: 你知道需求定律是什么吗？
用户: 不知道
AI: 那你觉得价格和需求量是什么关系？
用户: 不知道
AI: 好，那让我换个方式问——如果奶茶涨价了，你会怎么做？
```

**区别**：好的节奏先给信息建立基础，再用问题深化理解。坏的节奏对空白知识反复追问。

### 原则 3：每完成一个 KP 给明确的"收获感"

**问题**：学完一个知识点后没有任何反馈 = 感觉什么都没发生。
**解法**：每个 KP 教学结束时给一个简短的总结卡片。

```
✓ KP 1 完成：需求定律

  核心要点：价格↑ → 需求量↓（一般商品）
  你的表现：一次答对，理解清晰
  
  [继续学习 KP 2 →]
```

**赋予进展效应 (Endowed Progress Effect)**：让用户感觉已经有进展——进度条从 1/5 变成 2/5 比从 0 到 1 更有动力。

### 原则 4：教学法切换本身就是"变化"

**问题**：连续 25 轮同样节奏的对话 = 单调。
**解法**：5 种知识类型 × 5 种教学法 = 自然变化。

```
KP 1（事实型）：AI 直接讲 + 你复述      → 快节奏，2 轮
KP 2（概念型）：AI 类比 + 你追问为什么    → 中节奏，4 轮
KP 3（过程型）：AI 演示 → 你做一遍       → 动手感，5 轮
```

教学法的切换本身带来节奏变化——不需要额外的"游戏化"来防止无聊。

### 原则 5：让用户控制节奏

**问题**：系统强制走完 5 步教学 = 被控制感。
**解法**：给用户"我已经会了，跳过"的选项（但要验证）。

```
AI: 接下来我们学"均衡价格"。你已经了解这个概念吗？

选项 A: "不了解，教我"  → 走完整教学流程
选项 B: "有点印象"      → 走一个快速确认问题，答对跳过，答错进教学
选项 C: "我很熟悉"      → 走一个挑战性验证问题，答对跳过，答错进教学
```

**关键**：不是真正的"跳过"——B 和 C 都有验证。但用户感觉自己在掌控节奏。

### 原则 6：先分块后交错

**问题**：连续教 5 个同类型 KP = 单调。交替教不同类型 = 没有图式建立。
**解法**：先连续教 2-3 个同类型（建立图式），再混合不同类型（交错练习）。

```
Block 1: KP 1（概念型）→ KP 2（概念型）→ KP 3（概念型）
         [连续 3 个概念型，建立"概念理解"的思维模式]
         
交错: 混合小测——概念 + 之前学的事实型 + 过程型问题穿插
      [打破模式，强制切换思维]
      
Block 2: KP 4（过程型）→ KP 5（过程型）
         [连续 2 个过程型，建立"按步骤做"的思维模式]
```

交错练习的 Hattie effect size 虽然只有 0.21，但它的价值在于防止"幻觉掌握"——学生以为连续做对了就会了，交错后发现并不会。

## 两种模式的 UX 差异

### 教学模式（课件，只教不考）

```
上传课件 → KP 提取 → 选择模块 → 教学 Session → 完成

教学 Session:
├── 进度条（3-5 个 KP 为一组）
├── 每个 KP: 轻量/完整教学对话
├── 每个 KP 完成: 收获卡片
├── 全部完成: Session 总结
└── 可选: "继续下一组" 或 "今天到这里"
```

**UX 重点**：轻松、快速获得理解。不施加压力。完成后可以直接走。

### 完整模式（教材书，教+考+复习）

```
上传教材 → KP 提取 → 选择模块 → 阅读原文 → 教学 Session → QA → 测试 → 复习

教学 Session（同上，但节奏可以更深入）
教学完成后：
├── "准备好进入 QA 了吗？" → QA（现有系统）
├── QA 全完 → 生成笔记 → "准备测试了吗？"
├── 测试（现有系统，80% 过关）
└── 过关 → 进入间隔复习
```

**UX 重点**：结构感、成就感。有明确的"通关"概念。进度追踪更重要。

## 具体 UX 组件建议

| 组件 | 用途 | 设计要点 |
|------|------|---------|
| **KP 进度列表** | 显示当前 session 的 KP 及完成状态 | 复用现有 KnowledgePointList 组件（done/active/pending） |
| **教学对话区** | AI-用户对话 | 复用 ChatBubble 组件；AI 消息用 AIResponse 渲染 |
| **收获卡片** | 每个 KP 完成后的总结 | 新组件：KP 名 + 核心要点 + 表现评价 |
| **Session 进度条** | 整体进度 | 复用 SegmentedProgress（correct=完成/current=当前/unanswered=待学） |
| **Session 总结** | 全部 KP 完成后 | 新组件：完成数 + 掌握情况 + 下一步建议 |
| **快速选择** | "我已经会了" 选项 | 简单按钮组，不是复杂 UI |

**复用现有组件**：SplitPanel（左侧 KP 列表 + 右侧对话区 + 底部反馈），与 QA session 布局一致，用户无需重新学习界面。

## 总结：让学不无聊的 7 条规则

1. **终点可见**：用户始终知道"还剩几个 KP"，不是无底洞
2. **先给再问**：先呈现信息建立基础，再追问深化，不要对空白知识反复逼问
3. **小胜驱动**：每完成一个 KP 给明确反馈（收获卡片），不是学完 5 个才有反应
4. **自然变化**：教学法切换带来节奏变化，事实型快、评估型深，不是均匀节奏
5. **用户掌控**：让用户选择节奏（"教我"/"快速确认"/"我已经会了"），但暗中验证
6. **短 session**：每次 10-20 分钟 / 3-5 个 KP，不要一口气教完整个模块
7. **复用已知界面**：教学界面复用 SplitPanel + KP 列表 + ChatBubble，降低认知负荷
