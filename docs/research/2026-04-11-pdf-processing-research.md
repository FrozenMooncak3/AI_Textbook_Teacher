- **保底**: PaddleOCR v4 升级（当前版本）— 最快、中文最好、Apache 2.0，但需自建结构化逻辑

## 三、文字版 PDF 提效 + 智能处理策略

### 1. 文字版 PDF：pymupdf4llm 升级

**发现**: 当前 `ocr_server.py` 用 `page.get_text()` 只提取纯文字。**pymupdf4llm** 是 PyMuPDF 的薄封装层，输出结构化 Markdown（保留标题、表格、列表、粗体斜体），速度 ~0.12 秒/整个文档（比当前还快）。

**好处**: 
- KP 提取质��提升（AI 能看到标题层级、表格结构）
- text-chunker 的标题检测更可靠（标题已经是 `#` Markdown 格式）
- 几乎零开发成本（同引擎，换调用方式）

### 2. 混合 PDF 智能检测

当前代码（ocr_server.py line 116-126）: `if text:` 就跳过 OCR。问题：阈值太低，有些扫描 PDF 重新保存后有少量残余文字。

**改进伪代码**:
```python
def classify_page(page):
    text = page.get_text().strip()
    char_count = len(text)
    images = page.get_images()

        return "scanned"    # 需要 OCR
    else:
        return "mixed"      # 提取文字 + OCR 图片区域
```

工具参考: `ocr-detection` PyPI 包可做页面级分类（含置信度）。

### 3. 渐进式处理架构

**当前**: 上传 → OCR 全部页面(阻塞) → 用户才能开始

**改进**:
```
上传 PDF
  ↓
[即时] 分类所有页面 (< 1s)
  ↓
[Phase 1 - 秒级] ���取文字页 → 按章节分块 → 模块 1 文字就绪 → 用户可开始阅读
  ↓
[Phase 2 - 后台] OCR 扫描页 → 每完成一个模块的页面，解锁该模块
  ↓
[Phase 3 - 后台] 逐模块 KP 提取 → 模块逐步解锁教学/QA
```

**需要的改动**: 
- DB 新增模块级状态 `module_status: pending | text_ready | ocr_done | kp_ready`
- text-chunker 支持部分文本输入（先处理文字页，OCR 页完成后插入）
