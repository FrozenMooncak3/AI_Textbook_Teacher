# 两种学习模式：教学模式 + 完整模式

**日期**: 2026-04-11
**类型**: idea
**状态**: open
**来源**: 用户提出，基于月饼投资教学 spec 分析后的产品洞察

---

## 核心想法

加入教学环节后，产品自然分为两种学习模式：

### 模式 1：教学模式 (Teaching Mode)
- **适用**: 课件/PPT/讲义 — 信息密度高但没有展开解释的内容
- **流程**: 上传 → KP 提取 → AI 教学（按知识类型匹配方法）→ 结束
- **不含**: 正式 QA 出题、测试、复习循环
- **用户场景**: 学生有课件但听不懂课 → 上传课件 → AI 把每个知识点用合适的方式讲一遍
- **价值**: 补充课堂教学的不足，不替代课程本身的考核

### 模式 2：完整学习模式 (Full Learning Mode)
- **适用**: 教材书 — 需要深度理解+长期记忆的内容
- **流程**: 上传 → KP 提取 → 阅读原文 → AI 教学 → QA → 测试（80%过关）→ 间隔复习
- **含**: 完整的教学+考核+复习循环
- **用户场景**: 学生要学一本教材 → 需要系统化地学会并记住
- **价值**: 确保深度理解+长期保持

## 为什么这个想法重要

1. **解决了课件 vs 教材的产品难题**: 之前一直烦恼课件需要不同的学习模式，现在"只教不考"的模式天然适合课件
2. **降低了使用门槛**: 教学模式更轻量，用户可以快速获得价值（"帮我讲懂这个"），不需要承诺完整学习循环
3. **扩大了使用场景**: 
   - 课前预习 → 教学模式
   - 系统学习 → 完整模式
   - 考前突击 → 看用户选择
4. **用户可以自己选择**: 不是系统判断，而是用户根据自己的需求选

## 与 MVP 扩展的关系

- 教学模式的前提是"教学环节"的实现——这是 MVP 扩展的核心
- 完整模式 = 现有流程 + 教学环节
- 两种模式共享同一套教学引擎，只是后续环节不同

## 设计考虑

- 用户选择时机：上传后？还是进入模块时？
- 模式切换：课件用户后续想做题了，能否切换到完整模式？
- KP 提取方式：课件的 KP 提取策略可能和教材书不同（更短、更密集）

## 去向

纳入 MVP 大改 brainstorm——作为教学系统设计的产品框架。
