## 决策 + 为什么

### 产品定位：教练而非工具
- **选了**：教练定位——系统告诉用户该做什么，用户只负责执行
- **否决了**：工具定位（竞品路线：大纲、翻译、划词、错题本，所有动作由用户发起）
- **为什么**：目标用户自驱不够，"教练"比"工具"值钱。竞品全是工具，没有教练。对于需要外力推动的用户，教练解决的是核心问题，工具只是锦上添花

### 目标用户：中国留学生中"想学但自驱不够"的群体
- **完整推导链**：不是纯学霸（不需要教练）→ 不是纯来玩的（不会用学习产品）→ 是中间那批：离开国内中学和家长约束后，丧失了外力结构，想学但推不动自己。与创始人自身痛点一致
- **为什么不是其他用户**：学霸不需要，纯混子不付费。这群人有付费意愿（家里有钱送留学）且有真实痛点
- **否决了**：泛化的"任何学习者"——太宽，产品没法聚焦

### MVP 范围选择：教练核心 + 阅读器 + 独立问答
- **选了方案 B**（教练优先 + 阅读器 + 独立 Q&A）
- **否决方案 A**（阅读器优先）：阅读器不是核心价值，先做阅读器等于先做皮肤后做骨头
- **否决方案 C**（全做）：范围太大，MVP 要快
- **为什么 B**：教练链路是产品灵魂，先让灵魂跑通，再穿衣服

### 5 个 AI 角色分工
- **为什么不用一个 AI**：CC 的经验证明，一个 AI + 800 行 prompt + 一个上下文窗口会遗忘规则、混淆上下文、提问污染学习流程
- **核心 insight**：CC 的问题不是 prompt 质量不够（用户的 v2.0 prompt 已经很好了），而是**平台限制**——上下文窗口是有限的公共资源，什么都往里塞必然出问题。Web App 可以给每个 AI 角色独立的、聚焦的上下文

### Q&A 改为即时反馈
- **选了**：每题答完立即显示评分和解析
- **否决了**：全部答完后批量反馈（旧方案）
- **为什么**：(1) Ericsson 即时反馈原则；(2) 旧方案是 CC CLI 的技术限制导致的妥协，Web App 无此限制；(3) 即时反馈的短循环对 ADHD 用户更友好

## 用户 insight

- "我觉得 MVP 一定是先解决我这种类型的人的痛点"——创始人即目标用户，先解决自己的问题
- "CC 的核心问题不是 prompt 质量，而是上下文管理"——区分了 A（prompt 质量）、B（平台限制）、C（流程维护）三个痛点，B 和 C 是 Web App 天然解决的
- "竞品全是工具，没有一个是教练"——用户逐一分析了 NotebookLM、Kimi、各种 AI 学习工具，发现都是用户主动发起动作，没有一个系统主动 push 用户
- "最终愿景是个性化 CLI / 外置大脑 / RAG 系统"——MVP 是第一步，长期目标更远
- 用户有 ADHD——这是很多设计决策的隐性驱动力：即时反馈、短循环、系统替你决定、减少决策疲劳
- 用户有一个 800+ 行的 v2.0 学习系统 prompt，引用了 Gagné、Sweller、Bloom、Roediger、Ericsson、Woźniak SM-2 等学习科学理论。这个 prompt 是整个产品设计的源头
