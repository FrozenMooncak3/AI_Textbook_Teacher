## 2026-04-23 更正（决定性证据）

**原诊断 waitUntil / fire-and-forget 假设完全错误。** 真根因是 **Cloud Run 容器内存耗尽 kernel OOM kill**。

取自 Cloud Run logs（2026-04-22T11:25-11:40 UTC，服务 `ai-textbook-ocr`）：

```
11:25:31.658Z [INFO]    HTTP POST .../ocr-pdf → 202 (0.898s)   ← Vercel 请求已到 Cloud Run
11:25:32.466Z [DEFAULT] 169.254.169.126 "POST /ocr-pdf" 202    ← Flask 返 202
11:25:38.055Z [DEFAULT] google.api_core FutureWarning           ← Vision lib import
11:26:27.557Z [DEFAULT] google.cloud.vision_v1 FutureWarning    ← 继续导入
（13 分钟静默 — OCR 在跑但 Python 无 progress log）
11:40:49.882Z [DEFAULT] Out-of-memory event detected in container
11:40:49.934Z [WARNING] Container terminated on signal 9
```

**证据推翻原 waitUntil 假设**：Cloud Run **确实收到了** `/ocr-pdf` 请求并返了 202。Vercel 侧 fire-and-forget fetch 根本没丢。问题完全在 Cloud Run 内部，处理了 ~15 分钟后被内核 OOM killer SIGKILL。

## 真根因

`scripts/ocr_server.py:137-152` `google_ocr()` 每页新建一个 `vision.ImageAnnotatorClient()`：

```python
def google_ocr(page_image):
    from google.cloud import vision
    client = vision.ImageAnnotatorClient()   # ← 369 页 × 每页新建
    ...
```

每个 client 包含 gRPC channel + TLS cert pool + auth token cache。Python GC 懒回收，369 个 client 残骸 + 每页 PIL Image (1.5x ~2-4 MB) + PNG buffer 累积占用 heap。

Cloud Run 默认 container memory = 512 MiB，14+ 分钟后撞线，kernel OOM kill → Python 进程根本没机会打错误日志（SIGKILL 绕过用户态），callback 永远不会触发 → `modules.ocr_status=pending` 永久 stuck。

## 为什么之前没暴露

- M4.5 book 5 smoke（6 页）：页数少，client 残骸不够撞 512 MiB
- book 7/10/11/12：也是小书，小样本下容器 OOM 前就已完成
- book 13 是第一本真正大书（369 页），第一次稳定触发 OOM

## 实际修复（M4.6 T15，2026-04-23）

派发给 Codex，commit `e0bb0c5`：

1. **Vision client 模块级 singleton** —— `_get_vision_client()` 带双重检查锁，懒初始化，全进程复用一个 client。`google_ocr` 调用替换。消除每页 client 残骸
2. **显式 gc** —— `process_pdf_ocr` 循环内每 50 页 `gc.collect()`，主动清 PIL image + PNG buffer
3. **进度日志** —— 每 50 页 print `[ocr] book {id} progress {n}/{total}`，flush=True。下次出问题 Cloud Run log 有线索

**并行修复（用户 Console 操作）**：Cloud Run memory 512 MiB → 4 GiB。5 分钟 Console 配置，部署新 revision。

## 经验教训

**原诊断为什么跑偏**：看到"Cloud Run 侧无日志"就假设"请求没送达"，忽略了一个基础事实——"没日志"≠"没处理"。Python 进程被 SIGKILL 时来不及打日志。正确的第一步应该是**先拿到 Cloud Run 生命周期事件**（OOM / SIGKILL / instance scaled down），再判断进程状态。

后续教训 → `feedback` memory 已有 `feedback_diagnose-first.md`（Always logs/DB query before fix）；本次强化：对跨服务链路，**两端日志都要查清楚**才能下结论，只看 Vercel DB logs 会错判为外部问题。
