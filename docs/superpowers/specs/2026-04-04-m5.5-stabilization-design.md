# M5.5 Design Spec — Stabilization & App Shell

> **Goal**: Make the app navigable, stable, and feel like a real application.
> **Acceptance criteria**: User can click anywhere, navigate freely, never hit a white screen or dead end. All loading states communicate what's happening.

---

## 1. Sidebar Navigation (Highest Priority)

The sidebar is the app shell — the persistent frame that makes this feel like a real application, not a collection of web pages. **This must be production-quality.**

### 1.1 Layout Architecture

```
┌──────────────┬──────────────────────────────────┐
│              │                                    │
│   Sidebar    │         Content Area               │
│   (fixed)    │    (only this region changes)       │
│              │                                    │
│   ~240px     │         remaining width             │
│   desktop    │                                    │
│              │                                    │
└──────────────┴──────────────────────────────────┘
```

- Sidebar is rendered in the **root layout** (`src/app/layout.tsx`) so it persists across all page transitions.
- Content area scrolls independently; sidebar has its own scroll if content overflows.
- Page navigation only replaces the content area — the sidebar never re-renders on navigation.

### 1.2 Sidebar Content Structure

The sidebar content is **context-aware** — it adapts based on the current route.

**Layer 1 — Global (always visible):**
```
首页
上传教材
```

**Layer 2 — Book-level (visible when inside /books/[bookId]/**):**
```
── [Book Title] ──────────
阅读原文          → /books/[bookId]/reader
模块地图          → /books/[bookId]/module-map
仪表盘            → /books/[bookId]/dashboard
错题本            → /books/[bookId]/mistakes
── 模块列表 ──────────
模块 1: [title]   (学习状态)
模块 2: [title]   (学习状态)
...
```

**Layer 3 — Module-level (visible when inside /books/[bookId]/modules/[moduleId]/**):**

The current module expands to show sub-pages:
```
模块 3: [title]   ← expanded
  学习            → /books/[bookId]/modules/[moduleId]
  Q&A             → /books/[bookId]/modules/[moduleId]/qa
  测试            → /books/[bookId]/modules/[moduleId]/test
  错题            → /books/[bookId]/modules/[moduleId]/mistakes
```

**Learning status labels** (must include ALL 6 states from architecture.md):
| Status | Label |
|--------|-------|
| unstarted | 未开始 |
| reading | 阅读中 |
| qa | Q&A 中 |
| notes_generated | 笔记已生成 |
| testing | 测试中 |
| completed | 已完成 |

Other modules in the list remain collapsed (clickable to navigate).

**Review entry point**: Reviews are accessed from the homepage "待复习" list (already exists). NOT duplicated in the module sidebar — reviews are schedule-driven, not module-initiated.

### 1.3 Behavior

| Behavior | Specification |
|----------|--------------|
| Active page | Highlighted with background color + left accent bar |
| Desktop (≥1024px) | Sidebar always visible, default expanded (~240px) |
| Desktop collapse | Toggle button to collapse to icon-only mode (~56px). State persisted in localStorage |
| Mobile (<1024px) | Sidebar hidden by default. Hamburger button (☰) in top-left opens sidebar as overlay with backdrop |
| Mobile close | Click backdrop, press ESC, or click ☰ again to close |
| Module list | Scrollable within sidebar if many modules. Current module auto-scrolled into view |
| Book title | Truncated with ellipsis if too long. Full title on hover tooltip |
| Transitions | Smooth slide animation for expand/collapse (200ms ease) |

### 1.4 Data Requirements

The sidebar needs to know:
- Current book (id, title) — derived from URL params
- Module list (id, title, order_index, learning_status) — fetched via lightweight API call or passed from layout

**Recommended approach**: Create a `SidebarDataProvider` context in the book-level layout (`src/app/books/[bookId]/layout.tsx`) that fetches book + module list once and provides it to the sidebar. This avoids re-fetching on every page change within the same book.

### 1.5 Visual Standards

- Clean, minimal design consistent with existing Tailwind + gray/blue palette
- Font sizes: section headers 11px uppercase tracking-wide, nav items 14px
- Hover states on all interactive items
- Smooth transitions, no janky re-layouts
- Must look professional — reference apps: Notion sidebar, Linear sidebar, Vercel dashboard

### 1.6 File Changes

| File | Change |
|------|--------|
| `src/app/layout.tsx` | Add SidebarLayout wrapper |
| `src/app/books/[bookId]/layout.tsx` | New — provides book/module context to sidebar |
| `src/components/Sidebar.tsx` | New — main sidebar component |
| `src/components/SidebarProvider.tsx` | New — context for sidebar state (expanded/collapsed, mobile open/close) |
| All existing page components | Remove any ad-hoc back buttons or navigation links that the sidebar replaces |
| Pages using `h-screen` / `min-h-screen` | Change to `h-full` / `min-h-full` to fit within the sidebar layout's content area. **Critical for PdfViewer** which uses `h-screen` in 3 places (main container, loading state, error state) — will overflow with sidebar if unchanged |

## 2. Bug Fixes

### 2.1 Auto-trigger KP Extraction After OCR

**Problem**: When OCR completes (`parse_status='done'`), `ProcessingPoller.tsx` just calls `router.refresh()`. User has to manually discover and click "生成模块地图" button.

**Fix**: After `ProcessingPoller` detects `parse_status='done'`:
1. Automatically call `POST /api/books/[bookId]/extract` to trigger KP extraction (NOT `POST /api/modules` — that's the old module generation endpoint)
2. Switch display to "正在提取知识点..." with stage label
3. Poll `kp_extraction_status` until complete, then refresh

**⚠️ Response structure**: The status API uses `handleRoute` wrapper. Implementer MUST verify the actual JSON response shape — it may be `{ success, data: { parse_status, ... } }` rather than `{ parse_status, ... }` at the top level. Check `src/lib/handle-route.ts` and test with a real request before coding the polling logic.

**Flow after fix**:
```
Upload PDF → OCR (real progress: 第X/Y页) → KP提取 (stage label) → Module map appears
```

User never has to click anything after uploading.

**File changes**:
| File | Change |
|------|--------|
| `src/app/books/[bookId]/ProcessingPoller.tsx` | Add auto-trigger logic after OCR completes, add KP extraction stage display |

### 2.2 Upload Progress Display

**Problem**: ProcessingPoller shows generic "处理中" spinner. Backend has `ocr_current_page` / `ocr_total_pages` in the status API response.

**Fix**: Display real OCR progress from status API:
- During OCR: `正在识别第 12/45 页` + real progress bar
- During KP extraction: `正在提取知识点...` (stage label, no bar — no numeric progress available)
- On error: Show error message with link to logs

**File changes**:
| File | Change |
|------|--------|
| `src/app/books/[bookId]/ProcessingPoller.tsx` | Read ocr_current_page/ocr_total_pages from status response, render progress bar + stage labels |

### 2.3 White Screen Investigation & Error Boundaries

**Problem**: Some navigation transitions result in white/blank screens.

**Investigation scope**: Gemini must test every page transition and identify which routes crash. Likely causes:
- Server component data fetch failures (DB query returns null → unhandled)
- Invalid route params (deleted book/module IDs)
- Missing error boundaries

**Fix**:
1. Add `error.tsx` error boundary to every route group:
   - `src/app/error.tsx` (root)
   - `src/app/books/[bookId]/error.tsx`
   - `src/app/books/[bookId]/modules/[moduleId]/error.tsx`
2. Each error boundary shows a user-friendly message + "返回首页" link
3. Audit all server components for unhandled null/undefined cases
4. Add `not-found.tsx` pages for invalid IDs

**File changes**:
| File | Change |
|------|--------|
| `src/app/error.tsx` | New — root error boundary |
| `src/app/books/[bookId]/error.tsx` | New — book-level error boundary |
| `src/app/books/[bookId]/modules/[moduleId]/error.tsx` | New — module-level error boundary |
| `src/app/not-found.tsx` | New or update — global 404 page |
| Various server components | Audit and add null checks |

## 3. Loading State Standards (Global UX Rule)

**Principle**: Every loading state must communicate what's happening. No bare spinners.

| Scenario | Display |
|----------|---------|
| PDF OCR | Real progress bar + `正在识别第 X/Y 页` |
| KP extraction | Stage label: `正在分析文本...` → `正在提取知识点...` → `正在生成模块...` |
| QA question generation | Stage label: `正在生成练习题...` |
| Test question generation | Stage label: `正在生成测试题...` |
| Review question generation | Stage label: `正在生成复习题...` |
| Notes generation | Stage label: `正在生成学习笔记...` |
| Screenshot OCR | Stage label: `正在识别文字...` |
| AI feedback / scoring | Streaming text output (already implemented ✅) |

**Implementation**: Create a shared `LoadingState` component that accepts:
- `type`: 'progress' (with bar) or 'stage' (label only)
- `current` / `total`: for progress type
- `label`: stage description text

All existing loading placeholders across the app must be replaced with this component.

**File changes**:
| File | Change |
|------|--------|
| `src/components/LoadingState.tsx` | New — shared loading state component |
| Various page/component files | Replace bare spinners with LoadingState |

## 5. error_type Normalization (Tech Debt)

**Problem**: `review/respond` has `normalizeReviewErrorType()` for fuzzy matching AI error types to the 4 standard values (`blind_spot`, `procedural`, `confusion`, `careless`). `test/submit` lacks this — AI returns non-standard values that go directly into DB.

**Fix**: `normalizeReviewErrorType()` already exists in `src/lib/review-question-utils.ts`. Simply import and reuse it in `test/submit`. Do NOT create a new file.

**File changes**:
| File | Change |
|------|--------|
| `src/app/api/modules/[moduleId]/test/submit/route.ts` | Import `normalizeReviewErrorType` from `@/lib/review-question-utils` and use it instead of raw `?? 'blind_spot'` fallback |

## 8. Acceptance Criteria

1. **Navigation**: User can reach any page from any page using the sidebar. No dead ends.
2. **No white screens**: All routes have error boundaries. Invalid IDs show friendly error, not blank page.
3. **Upload flow**: Upload PDF → see real OCR progress → KP extraction auto-starts → modules appear. Zero manual steps.
4. **Loading states**: Every loading state shows what's happening (progress bar or stage label). No bare spinners.
5. **Screenshot AI**: Dialog can be closed. Shows captured image and OCR text.
6. **error_type**: All mistakes records use one of 4 standard values.
