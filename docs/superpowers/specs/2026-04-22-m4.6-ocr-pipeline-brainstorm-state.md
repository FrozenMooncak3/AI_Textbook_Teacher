## 基础设定（不会变）

### 触发背景
- M4.5 收尾 live E2E：book 11 / book 12 上传后 confirm 成功 → fire-and-forget `runClassifyAndExtract` → fetch `${OCR_SERVER_URL}/classify-pdf` 超时 4-6 分钟报 `TypeError: fetch failed`
- OCR 服务直接外部 curl 探测 4.5 秒返 200 正常 → 服务本身没问题
- 瓶颈在 Vercel 函数出站方向（候选：google-auth-library token 获取 hang / Vercel↔GCP 网络抖动 / Vercel 函数运行时 bug）
- 现状：生产无法上传任何扫描 PDF；book 10/11/12 是 stuck artifacts

### 调用链（已确认）
1. `POST /api/books/confirm` → HeadObject 校验 → UPDATE books → `void runClassifyAndExtract(bookId, objectKey)`（fire-and-forget）
2. `runClassifyAndExtract` 顺序 await 3 个 Cloud Run 端点：
   - `POST ${OCR_SERVER_URL}/classify-pdf`（小，分类页面类型）
   - `POST ${OCR_SERVER_URL}/extract-text`（返回全文）
   - `POST ${OCR_SERVER_URL}/ocr-pdf`（fire-and-forget，扫描页 OCR）
3. **每次** fetch 前都跑 `buildOcrHeaders(url)`：
   - `cachedAuth: GoogleAuth` 单例（首次 `JSON.parse(GCP_SA_KEY_JSON)`）
   - `auth.getIdTokenClient(audience)` ← **每次重新构建 client，未缓存**
   - `client.getRequestHeaders()` ← 内部去 OAuth2 token 端点换 ID token（google-auth-library 内部缓存到 client 实例上，但 client 每次新建 → 等于无缓存）
4. `fetch()` 无 timeout 设置 → 一旦 hang 走到 Vercel 函数 maxDuration（Fluid 300s）

### 已尝试（2026-04-22）
- **Phase A**：同 commit redeploy（`dpl_C7HDhBS8gQsJZwAbeqfDXtupKziB` PROMOTED）测"Vercel 平台抽风"假设 → 未 live 验证
- **Phase B**：`OCR_REQUIRE_IAM_AUTH=false` + redeploy → 直接探测 Cloud Run 服务侧硬要求 IAM invoker，无 Bearer 1.3s 内 GCP 平台层 403 → 路径不可行，env 已恢复 `true`

### 不可破红线（产品 / 架构）
- Cloud Run OCR 服务 IAM invoker 硬要求不能改（GCP 平台层强制）
- Vercel Hobby tier，runtime logs 看不到内部 trace（无 paid log drain）
- Vercel Hobby Fluid 300s 函数超时硬上限
- M4.5 已落地 presigned URL 直传 R2 + confirm fire-and-forget（4.5MB body 限制已绕过，不再讨论）
- 内部信号护城河仍生效（`kp.type/importance/detailed_content/ocr_quality` 不暴露）

## 已拍死的决策（不再讨论）

### 决策 1：M4.6 ambition level（2026-04-22 拍板）
**选择**：A+（修 hang + 加保命 + 清旧）。
- 修 hang：必须，阻塞性问题
- 加保命：fetch 30s timeout + 1 次自动重试，避免再卡 5 分钟没反馈
- 清旧：清理 book 10/11/12 stuck artifacts 不让 DB 脏
- **不做** Cloud Run perf（min-instances / Vision batching）：月费 + 当前用户量收益不清
- **不做** 自动报警：属 Phase 3 云部署收尾

**Rationale**：用户明确反馈 "现在链路完全跑不通" 是阻塞性问题，先紧后松。Perf / alerting 都是 nice-to-have，等线复活了再评估。

**拒绝替代**：
- 纯 A（仅修 hang）— 排除，因为不加 timeout/retry 下次抽风又是 5 分钟黑屏
- B（A+ Cloud Run perf）— 排除，月费 + 收益不清，留给 5/6 月用户量上来再决策

### 决策 2：诊断 vs Workaround 策略（2026-04-22 拍板）
**选择**：先 workaround，不等调研。直接打两拳（缓存 IdTokenClient + fetch timeout/retry）→ 部署 → 用户测试。修好则收尾，没好才进决策 3 诊断阶段。

**Rationale**：
- 两个 workaround 都有强嫌疑代码证据（getIdTokenClient 每次新建 = 明显 anti-pattern；无 timeout 等于死等）
- 都是小改 + reversible（git revert 一行 / 调超时阈值），失败成本低
- 用户 explicit 表达紧急情况要绕开标准流程
- 标准流程的 🔴 research 适合"难反悔决策"；这里反悔成本远低于 1 周调研代价

**拒绝替代**：
- 纯诊断先行 — 排除，得加 instrumentation + 等下次 hang 复现 + 看 logs，至少 3-4 天才能拿到数据
- 并行（一边 workaround 一边研究）— 排除，过度复杂，workaround 一次部署就能验证嫌疑

### 决策 4：Workaround 具体方案（2026-04-22 拍板）
**选择**：
- (a) `src/lib/ocr-auth.ts` 加 `clientCache: Map<string, IdTokenClient>` 按 audience 缓存，永久缓存（client 自管 token TTL）
- (b) `src/lib/upload-flow.ts` 每次 fetch 加 `AbortSignal.timeout(30_000)` + 复用 `src/lib/retry.ts` 的 `retryWithBackoff` 重试 1 次（退避 1s）

**Rationale**：两条改动都基于已读代码的精确定位，最小侵入。

**拒绝替代**：
- 改 Edge runtime — 排除，需调研 google-auth-library 在 Edge 兼容性，超出当前快修范围
- 用 Cloud Tasks / Pub/Sub 异步队列 — 排除，架构改造大，与 A+ 范围矛盾

### 决策 6：book 10/11/12 stuck artifacts 处置（2026-04-22 拍板）
**选择**：删 DB 行（cascade 删 modules），保留 R2 文件，写 `docs/journal/2026-04-22-stuck-books-snapshot.md` 备忘当时状态。

**Rationale**：DB 行删了让查询干净；R2 文件留着方便日后重传诊断；备忘是诊断 trail。

**拒绝替代**：
- 全删（含 R2）— 排除，重传成本 + 失去诊断样本
- 全留 — 排除，DB 脏污染 books 列表 UI

### 决策 7：Live 测试协调机制（2026-04-22 拍板）
**选择**：异步——Claude 准备 deploy + `docs/m4.6-test-checklist.md`（含步骤 + 命令），用户上线时复制跑，截图 / 文字反馈。

**Rationale**：用户不固定在电脑前，同步等成本高；异步把准备工作做完，用户的时间只用在测 + 反馈。

**拒绝替代**：
- 同步等用户 — 排除，效率太低
- cron 预约 — 排除，过度工程，用户日程不规律
