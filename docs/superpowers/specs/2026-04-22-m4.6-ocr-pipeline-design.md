## 摘要

**问题**：M4.5 收尾 live E2E 测试暴露——14MB 真书上传 → confirm 成功 → fire-and-forget OCR 调用 hang 4-6 分钟，`TypeError: fetch failed`。OCR 服务本身正常（外部直探 4.5s 返 200），瓶颈在 Vercel 函数出站方向。结果：生产无法处理任何扫描 PDF。

**大白话**：用户传 PDF → 上传成功 → 准备页一直转圈不动。后台是"网站给 OCR 服务打电话"那一步，电话拨出去接不通，4-6 分钟才放弃。OCR 服务本身没坏（直接绕开网站打过去 4.5 秒就接了）。

**整体方案**：先打两个最有把握的 workaround 拳（缓存 IdTokenClient + fetch timeout/retry），不等深度调研。一次部署 + 一轮 live 测试验证。打中即收尾；没打中再进诊断阶段加 instrumentation。

**范围边界（A+ 范围）**：
- 修 hang（必须）
- 加保命机制（fetch 30s timeout + 1 次自动重试）
- 清理 book 10/11/12 stuck artifacts
- **不做** Cloud Run perf 调优（min-instances / Vision batching）— 月费 + 不知道值不值
- **不做** 自动报警系统 — 属 Phase 3 云部署收尾

预计 2 天内（workaround 修好 case），最差 1 周（要进诊断阶段加日志深挖）。

## 背景与触发

### 触发链
- 2026-04-22 M4.5 收尾 live E2E 重跑 book 11 / book 12 失败两次
- `runClassifyAndExtract` → `fetch ${OCR_SERVER_URL}/classify-pdf` 卡 4-6 分钟
- OCR 服务直接外部 curl 探测 4.5 秒返 200 → 服务侧正常
- 根因在 Vercel 函数出站 → Cloud Run 这一段

### 已尝试不可行
- **Phase A** 同 commit redeploy（`dpl_C7HDhBS8gQsJZwAbeqfDXtupKziB` PROMOTED）测"Vercel 平台抽风"假设 → 未 live 验证
- **Phase B** `OCR_REQUIRE_IAM_AUTH=false` → 直接探测发现 Cloud Run 服务侧硬要求 IAM invoker，无 Bearer 1.3s 内 GCP 平台层 403 → 路径不可行，env 已恢复 `true`

### 关键代码发现（嫌疑 #1）

`src/lib/ocr-auth.ts:21-40` `buildOcrHeaders()` 每次 fetch 都重新跑 `auth.getIdTokenClient(audience)` + `client.getRequestHeaders()`。`cachedAuth: GoogleAuth` 单例存在，但 `IdTokenClient` 实例每次新建 → ID token 缓存形同虚设。一次上传走 3 次 fetch，等于 3 次 OAuth round-trip。

**大白话**：网站每次跟 OCR 服务说话之前都要先去 Google 那里换一张"通行证"。代码里换通行证的步骤每次都重做，没有"用完留着下次用"的存根。一次上传换 3 次。可能就是某次换通行证那一步卡住了。

## 工程实现清单

### OCR Client Refactor（决策 2 + 4 落地）

**`src/lib/ocr-auth.ts`** — 缓存 IdTokenClient
- 现：`cachedAuth: GoogleAuth | null` 单例 + 每次 `buildOcrHeaders` 重 `getIdTokenClient`
- 改：加 `clientCache: Map<string, IdTokenClient>`，按 `audience` key 缓存
- 逻辑：`if (clientCache.has(audience)) return cached; else build + cache`
- google-auth-library `IdTokenClient` 内部自动 refresh ID token（管自己的 TTL），所以 client 永久缓存安全

**`src/lib/upload-flow.ts`** — fetch timeout + retry
- 复用 `src/lib/retry.ts` 的 `retryWithBackoff` 包裹每次 fetch
- 每次 fetch 加 `signal: AbortSignal.timeout(30_000)`
- 重试 1 次（共 2 次尝试），退避 1s
- 重试前重新调 `buildOcrHeaders`（防 token 过期边界 case）
- 失败仍走 `markOcrFailure`，错误信息含 `timeout / aborted / retry exhausted` 明细

**大白话**：把"换通行证"的结果存起来下次直接用；打电话超过 30 秒不通就放弃挂掉重新打一次。

### Stuck Artifacts Cleanup（决策 6 落地）

- 一次性 SQL（写到 `scripts/cleanup-stuck-books-m4.6.sql`，跑后即可删）：先 `SELECT` 验证 user_id + 备份字段，再 `DELETE FROM books WHERE id IN (10, 11, 12) AND user_id = ...`
- modules 表通过 ON DELETE CASCADE 自动清掉
- R2 文件不动（用户日后重传同 PDF 还能用）
- 备忘：`docs/journal/2026-04-22-stuck-books-snapshot.md` 记 book id / title / file_size / parse_status / created_at / 失败摘要

### Test Protocol（决策 7 落地）

`docs/m4.6-test-checklist.md`：
- **准备**：登录 https://learn.zhouxin.uk
- **步骤 1（小 PDF baseline）**：上传 `sample-bug-report.txt` → 期望 60 秒内 firstModuleReady
- **步骤 2（14MB 真书）**：上传 14.2MB 教材 → 期望 60 秒内 firstModuleReady
- **步骤 3（失败时）**：截图 preparing 页（含 lastError + 时间戳）+ Vercel logs 第 1 屏 → 发给 Claude
- **期望**：步骤 1+2 都进入 firstModuleReady

### Acceptance Criteria（决策 8 落地）

- ✅ book 11 14MB 重传 → preparing 页 60 秒内 `firstModuleReady=true`
- ✅ 失败 case：30 秒内 `lastError` 字段明确（含 "timeout" 或 retry 信息）— 不是 5 分钟白屏
- ✅ Vercel function logs 含 "OCR call success" / "OCR call retry attempt 1" / "OCR call exhausted" 信号之一

## 风险与回滚

| 改动 | 风险 | 回滚路径 |
|------|------|---------|
| IdTokenClient 缓存 | 缓存的 client 持有过期 token 导致 401 | google-auth-library 自动 refresh，理论安全；万一不行 → revert + 加 1h TTL 强制重建 |
| fetch timeout 30s | 大 PDF classify 真的需要 >30s | 调大到 60s 或按端点分别配（classify 30s / extract 90s）— 看监控数据再调 |
| 自动重试 1 次 | 加重下游 OCR 服务负载 | 重试硬编码 1 次（不雪崩）；失败有 markOcrFailure 落 logs |
| stuck artifacts 删除 | 误删用户重要数据 | 先 SELECT 备份到 snapshot.md，再 DELETE；R2 文件全保留 |

无 env var toggle / kill switch（改动太局部，不值得加）。
